MILESING APPLICATION REALIZATION V0.2

PURPOSE
MilesingLight is one private native desktop application in which Milesing's
Light is executable mathematics.  Self numbers designate the operations that
act on authored work and exact measured wholes.  Win32 and DirectX expose the
local machine and portray the work; Milesing's Light supplies the named
mathematical meaning.

ONE NATIVE APPLICATION
APP.txt compiles the seven authored source bodies as one C++20 translation:

  1. SUPPORT.txt
  2. SYSTEMLOGS.txt
  3. 2DFRACTALIZER.txt
  4. 3DFRACTALIZER.txt
  5. EDITOR.txt
  6. RENDERER.txt
  7. APP.txt

The result is a 64-bit Windows desktop application using Win32, Direct3D 11,
Direct2D, DirectWrite, BCrypt, and Windows filesystem APIs.  The authored
source adds no server, account, telemetry, updater, cloud service, socket or
HTTP client, or public data collection.  File access begins only from a
deliberately selected path whose root is independently checked as a directly
attached local volume.  Open and Save As perform admission checks; every save
revalidates its target immediately before target inspection and writing.  UNC,
device, unknown-root, remote-volume, and any existing reparse-point component
are held.  A missing suffix is allowed only after every existing parent proves
direct and local.  This pathname validation is not handle-relative/no-follow
race elimination, so residual pathname TOCTOU remains.

SUPPORT.txt also compiles Generated/SELF_NUMBER_BINDINGS.inc.  That generated
data is an exact application surface of the binding Milesing's Light ledger:
112 self numbers times ten ordered fields, for 1,120 exact fields.  It remains
generated data inside the same native application rather than an additional
application body.

EXACT BINDING SHEET
In OPERATIONS, B opens the read-only binding sheet for the selected self number.
The sheet displays all ten exact fields from Generated/SELF_NUMBER_BINDINGS.inc,
ending with Direct proof.  Arrow keys navigate the 112 self numbers; the mouse
wheel and Page Up / Page Down scroll; Home and End move to the beginning and end;
B or Escape returns to operation execution.  Sheet interaction leaves operands,
expressions, receipts, invocation, and saved work unchanged.

LIGHT TREATMENT ALIVE NOW
LIGHT TREATMENT measures exact finite windows of the MilesingLight application,
authored world, explicitly admitted local machine, current process, renderer,
and explicitly active local workspace file.  Its one-time machine facts include

adapters admitted per window.  Measurement samples become
measured-bit wholes with source, duration, position, typed mathematical views,
and ancestry.  Light operations derive local graph wholes from them.  Finite
sheaf gluing then produces one global derived whole exactly when the local
wholes agree on every shared measured-bit key.

The completed LIGHT TREATMENT RelationValue then becomes the exact typed source
whole for MY LIGHT.  A separate MeasureWhole canonically encodes the finite unary
route converse.  DivinandWhole carries the source whole and route into the MY
LIGHT derived whole, retaining every source bit and every route bit in its exact
ancestry.  Its completed result can immediately become a typed source for a
later compatible derivation.

Application launch and local workspace opening create only a deterministic ready
association.  They capture no machine measurements.  A finite measure window
begins only through the explicit LIGHT TREATMENT controls below.

The direct gluing proof is constructive: every target key is covered; agreement
on overlaps makes its value independent of patch choice; restricting the merge
returns every local whole; and any other whole with the same restrictions must
agree at every target key.  A disagreement reports its exact key and produces no
global whole.

LIGHT TREATMENT CONTROLS

  F6       open the LIGHT TREATMENT workspace

The live status reports acquired occurrences, missed intervals, acquisition

only to that exact finite domain.  It is not a claim of omniscience.

WORKLOAD AND MACHINE CARE
Each acquisition records its QueryPerformanceCounter cost.  The adaptive cadence
targets at most one percent of the sampling interval for measurement work,
lengthening the interval to at least one hundred times the worst acquisition
cost observed in the active window.
Repeated measure families replace their preceding raw occurrence, missed
intervals remain explicitly missed, and live raw retention is capped at 256
families and 8 MiB.  Each occurrence selected for direct measured-bit expansion
is capped at 4096 encoded bytes.  Completed derivations release the raw
occurrences.
This permits increasingly rich recursive work while responding to the capacity
actually measured on the current machine.

THREE MEASURED AXES
The application keeps three different results distinct.  First, its operation
body expands losslessly into infinitely many lawful usable finite openings under
recursive Light operations.  Every admitted finite route retains the exact cause
and exact route needed for deterministic recalculation, whether or not every
result has already been portrayed.  Finite hardware evaluates one finite cut at
a time; that physical fact leaves the recursive family of usable openings
unbounded.  Second, selected measured-bit ancestry is exact for the bytes used
by the live completed whole.  Third, recurring external observations are
temporally coalesced by family, and the live status reports acquired bytes,
retained latest-family bytes, and their temporal reduction.  Released temporal
observations belong to that third-axis accounting and do not reduce the first
axis's retained causes and routes.  The 60-second measurement and evidence-based
60-minute projection are reported separately for these three axes.
