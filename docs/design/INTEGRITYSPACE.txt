INTEGRITY CONCEPT SPACE

STATUS
V0.2 hashes each complete canonical candidate and reopened work.  Per-chunk and
manifest hashing, embedded-resource evidence, package hashing and authenticated
encryption below remain expansion targets.

PURPOSE
Every durable work can prove which bytes were written, which exact Light and authored relations they represent, and whether reopening produced the same logical body.

LIVE V0.2 INTEGRITY CHAIN
- Form canonical version-2 text and parse it before writing any bytes.
- Hash the complete candidate text with BCrypt SHA-256.
- Write a sibling candidate, flush it, reopen it, compare its exact bytes and
  hash, and parse it again.
- Replace the destination while preserving a complete prior body when one
  exists; reopen the final path and repeat exact-byte, hash and semantic checks.
- Inventory prior/candidate siblings within explicit file-count and byte bounds;
  prune only verified old canonical priors and hold uncertain bodies.
- SHA-256 detects change.  It neither conceals the authored work nor proves who
  created it.

EXPANDED INTEGRITY TARGET
- Hash each uncompressed canonical chunk with BCrypt SHA-256.
- Record chunk type, generation, byte length, expanded ceiling, item ceiling, and hash in the manifest.
- Hash the canonical manifest separately and place its digest in the fixed header.
- Hash each selected embedded resource and record origin, license, and authored purpose.
- Before commitment, parse the complete candidate from a fresh read handle and compare its logical root, counts, relation routes, expressions, and hashes.
- After replacement, reopen the final path and compare again.
- Keep the protected prior body's independent manifest and digest.
- A migration writes a new candidate; it never edits the sole live work in place.
- Package integrity covers executable, embedded shaders/fonts/assets, documentation, licenses, and the empty network-capability declaration.

PROTECTION DISTINCTION
Any later confidentiality feature uses a separately chosen authenticated local
encryption and recovery route.  It remains distinct from current SHA-256 change
detection and is absent from v0.2.
