## The v0.2 realization

The catalog contains **112 self numbers**: the original 111 authored routes plus outer `LIGHT TREATMENT`, associated with `REALITY` without renumbering or relabeling the original map. Each entry is tied to ten exact binding fields:

1. SELF NUMBER
2. Operation
3. Codable symbol
4. Mathematical notation
5. Origin
6. Operands
7. Function
8. Reason
9. Actual C++ use
10. Direct proof

The generated binding table contains all **112 × 10 = 1,120** fields byte-exactly. The executable catalog checks its name, operation, codable symbol, signature, origin, extent, and authoritative source hash against that table. In the Operations workspace, `B` opens a read-only full-canvas binding sheet; arrows select a self number, the wheel and page keys scroll, and `B` or `Escape` closes it. Its controls are verified to leave operand drafts, receipts, and execution untouched.

The binding's Direct proof field and the executable fixture checks have distinct jobs. The binding displays Miles's exact mathematical statement. The fixture suite executes every current finite implementation route and reports **112/112** independently.

## Live operation and LIGHT TREATMENT

LIGHT TREATMENT is an explicit finite measurement and derivation workstation. Launch and workspace opening create a ready association while measurement remains dormant. In its workspace, `Q` begins a 60-second window, `D` begins a one-hour window, `Enter` begins a quick window or completes the active one now, and `Escape` revokes the active window and releases its ephemeral raw occurrences.

During an active window, MilesingLight measures its authored world, its own process and renderer, explicitly admitted local-machine facts, and the explicitly active local workspace file. Exact selected bytes become measured-bit whole selves with source, duration, spatial position, typed mathematical views, and ancestry. `SIZE` and `DESIGNING` derive recursive graph wholes. `LIGHT TREATMENT` applies finite sheaf gluing only when every shared measured member agrees, producing one exact global whole on the declared finite cover.

That completed global relation then enters **MY LIGHT** itself. A separately measured finite unary route names `converse`; `divinand(LIGHT TREATMENT global section, (converse))` constructs a live `DivinandWhole` retaining the exact source view, recursive derived-graph ancestry, route measure, duration, resulting relational converse, expression, receipt, and rooted spatial associations. The completed `DivinandWhole` can re-enter later compatible whole-self derivations.

Canonical version-2 workspaces preserve the associated LIGHT TREATMENT and MY LIGHT expressions, operand designations, results, and receipts. The richer live `DerivedGraphWhole` and `DivinandWhole` carriers remain runtime derivation bodies; this release does not claim that every in-memory carrier object or ephemeral observation is serialized.

## Local privacy and writing

The authored source adds no server, account, telemetry, updater, cloud service, socket, HTTP client, or public data collection. Machine/process/file measurement begins only through the explicit LIGHT TREATMENT controls. Raw measurement occurrences are bounded and ephemeral; completed expressions and receipts join the authored runtime.

`Ctrl+O`, `Ctrl+S`, and `Ctrl+Shift+S` are explicit local actions. `Ctrl+O` and Save As validate each dialog-selected path at admission; every save revalidates its target immediately before target inspection and writing. Validation admits directly attached local roots and walks every existing root-to-leaf component. Fixed, removable, optical, and RAM-disk roots are recognized; UNC, extended/device, unknown-root, remote-volume, and any existing reparse-point component are held. A missing suffix is allowed only after every existing parent proves direct and local. This pathname validation narrows exposure but is not handle-relative/no-follow race elimination, so residual pathname TOCTOU remains. Writing uses a flushed recovery candidate, protected prior, SHA-256 verification, semantic reopen, and bounded recovery inventory.

The final executable imports exactly `USER32.dll`, `ole32.dll`, `bcrypt.dll`, `d3d11.dll`, `dxgi.dll`, `d2d1.dll`, `DWrite.dll`, and `KERNEL32.dll`; `WS2_32`, `WINHTTP`, `WININET`, and `URLMON` are absent. PE inspection confirms x64 PE32+ Windows GUI, high-entropy VA, dynamic base/ASLR, NX/DEP, CFG instrumentation with a function-ID table, CET compatibility, `/GS`, an embedded `asInvoker` / `uiAccess=false` manifest, and a reproducible-build debug record. The certificate directory is empty; Windows reports `NotSigned`, and SignTool finds no signature.

Personal `.mlight` workspaces, machine data, private development instructions, internal notes, and third-party reference trees are absent from this repository.

## Strict version-1 migration

Version-1 input is admitted only when all 111 historical node self-number/operator/origin tuples match the exact published routes. For the eight routes whose explicit arity changed—MY LIGHT, NULLISM, ABSENTISM, RIGHTLY, TENSIONS, TRUST, CREST, and HOLA—the serialized historical result must exactly equal historical-route replay before deterministic enrichment and current replay. Every unchanged route requires a serialized result and canonical equality with current replay. Accepted input associates the new LIGHT TREATMENT body with REALITY and reserializes as stable canonical version 2. Missing or altered results, mismatched tuples, and malformed enrichment are rejected before live replacement.
